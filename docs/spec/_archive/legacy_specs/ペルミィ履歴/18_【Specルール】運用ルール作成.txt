## 1. Specファイルの運用

### 1.1 Specの位置づけ
- ペルソナアプリの仕様を決定する Spec ファイルを運用する。
- 仕様推論・設計・実装・回答は、常に**最新バージョンの Spec** を正とする。

### 1.2 Specファイル命名
- Specファイルは **`spec_v{バージョン}.md`** の形式で出力する。

### 1.3 出力前バックアップ（タイムスタンプ付与）
- Specを出力する前に、**現在のSpecファイルの末尾に日時のタイムスタンプを付与**し、バックアップ可能にする。
  - 例：`Last Updated (JST): YYYY-MM-DD HH:MM:SS +0900`
- タイムスタンプは必ず末尾に追記し、いつの状態か追跡可能にする。

### 1.4 最新Specの反映運用
- 出力された最新のSpecファイルは、ちょびが**手動でプロジェクトフォルダにアップロード**する。
- 全てのチャットは、アップロードされた**最新バージョンの Spec** を正として推論する。

### 1.5 「Spec出力」指示時のルール
- チャット内で **「Spec出力」** と指示された場合：
  1. 最新のSpecファイルと比較し、差分を洗い出す
  2. 更新内容を**整合性が取れる形に修整**して Spec を出力する
- **出力前に不整合があった場合は出力しない。**  
  その旨と不整合内容をちょびに伝える。

### 2.2 スマホモードの回答ルール
- 形式：**一問一答**

### 2.3 パソコンモードの回答ルール

    - 自動送信は絶対に行わない。

## 2. 優先順位（SSOT）
1. **spec_rule.md（本ファイル）**: プロジェクトの憲法。
2. **最新 Spec（spec_v{N}.md / spec_serverside_v{N}.md）**: 機能とデータの正解。
3. **最新 実装仕様（spec_serverside_dev_v{N}.md）**: 進行中の作業ログ。

## 4. 運用・実装・事故防止（継承ルール）

### 4.1 Spec運用
- 命名: `spec_v{N}.md`（Nは整数。上書き禁止・新version生成）。
- 差分出力: チャット内での修正ではなく、常に整合性を取った新versionファイルを生成・提示せよ。

### 4.2 モード別対応
- **スマホモード**: 1/2/3の選択形式による一問一答。
- **PCモード**: 実装・詳細設計。一度の質問は最大2項目まで。

### 4.3 実装とGit（破壊的操作の禁止）
- **コード提供**: 差し替え可能な「ファイル全文」または「完成形ユニット」で提供せよ（部分パッチ禁止）。
- **事故防止**: `git reset` 等の破壊的操作の前には必ず「事実取得・退避・単一コマンド提示」の三段階を踏むこと。
- **機密保持**: `.env` や秘密鍵の混入を厳格に監視し、`.gitignore` ではなく `git rm --cached` で対処せよ。
