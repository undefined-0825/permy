## 1. プロダクト概要と世界観

### 1.1 ゴールと目的（MUST）

- **最上位KGI**: ちょびさんの収益最大化。
- **中核課題**: 夜職ユーザーの返信コストと心理的負担（罪悪感）を、分身キャラ「ペルミィ」による代行提案で解決する。

### 1.2 コア価値（MUST）

- **ユーザーが主語**: AIは断定せず、常に3案提示し、ユーザーが選ぶ。
- **自動送信なし**: 必ずユーザーが手動で送信する。
- **夜職特化**: 本プロジェクトは夜職専用。ビジネス・標準モードは廃止し、夜職に完全シフトする。

### 1.3 世界観

- **キャラクター**: 魔法少女作品のサポートマスコット的な猫キャラ「ペルミィ」。
- **変身概念**: 診断に基づき「夜の自分」に変身することで、返信に対する心理的罪悪感を軽減する。

## 3. 返信生成仕様（MUST）

### 3.1 営業目的（コンボ）ID定義

- **10**: 来店確定
- **11**: 同伴獲得
- **12**: 休眠復活

### 3.2 A/B/C 案の役割（夜職固定）

- **A：余裕（信頼・安心）**: 最推奨。安定運用・失点最小
- **B：甘め（距離を縮める）**: 親密度向上
- **C：攻め（次アクション）**: 来店・同伴獲得を狙う攻勢案

### 3.3 生成ルール（C方式：構造固定＋文体可変）

- **要約排除**: 夜職モードでは要約を出力せず、3案のみを表示する。
- **基本骨格**: 「共感・前振り」→「状況確認/特別感」→「次アクション提案」。
- **品質**: 原則2〜4段落の長文（信頼構築）。短文が最適な文脈のみ短文を許容。
- **揺らし**: AI特有の同型連発を避けるため、構成や具体例を中程度に揺らす。
- **聞き返し**: 入力不足時は質問を1つに絞り、先に仮の3案を提示してから質問する。

### 3.4 禁止事項ゲート（中ゲート）

- 違法行為・脅迫等は **生成停止**。安全な断り方を提示。
- グレーゾーンは **安全寄せ補正**。最低1案はリスク回避案を必須とする。

# 付録Y: /generate followup（聞き返し）仕様（MUST）

**Added (JST):** 2026-03-01 18:25:00 +0900

## Y-1. 入力不足時の聞き返し（followup）レスポンス（SSOT / MUST）

/generate は通常の A/B/C に加え、**不足情報が1点だけある場合**に限り followup を同一レスポンスで返してよい（二段APIは禁止）。

### Y-1.1 条件（MUST）

- followup は **0または1件**。
- 質問は **1問のみ**、選択肢は **最大3**。
- followup が返った場合でも、A/B/C は必ず返す（UIを止めない）。

### Y-1.2 形式（json_schemaの出力仕様と一致させる）

```json
{
  "candidates": {
    "A": {"text": "..."},
    "B": {"text": "..."},
    "C": {"text": "..."}
  },
  "followup": {
    "key": "relationship_type",
    "question": "どんな関係性？",
    "choices": [
      {"id": "new", "label": "新規"},
      {"id": "regular", "label": "常連"},
      {"id": "vip", "label": "太客"}
    ]
  }
}
```
- `followup` は不要なら `null`。
- `key` は settings のフィールド名（例: `relationship_type` / `reply_length_pref` 等）。
- ユーザー選択後は `settings_snapshot` を更新して **次の生成**を行う（自動再生成はしない）。

- 既存データが `VisitPush`/`Flow` を持つ場合、サーバは `MamePush`/`Balance` に正規化して返す。


## AB-5. 生成の最重要ルール（MUST）

以下は生成時に必ず守る。fileciteturn38file0L100-L120

1) **履歴最優先（ミラーリング）**  
直近のトーク履歴（熱量、語彙、改行リズム、絵文字量）を強く模倣する。fileciteturn38file0L102-L107  

2) **呼称（呼び方）の動的判定**  
履歴から相手への呼び方を推測する。  
- 履歴内の第三者名（同僚/他キャスト等）との混同は **致命的NG**。  
- 判定不能な場合は **名前を呼ばない**。fileciteturn38file0L11-L15  

3) **時間整合性**  
現在時刻と最終送信時刻の差を考慮し、自然な挨拶（おはよ、まだ起きてる？等）を付与する。fileciteturn38file0L105-L107  

4) **人間味フィルタ**  
丁寧すぎる敬語/過剰な句読点/高い漢字率を排除。  
改行でリズムを作り、**140文字以内の短文（1吹き出し）**で完結させる。fileciteturn38file0L13-L15 fileciteturn38file0L106-L107  

5) **新規客対策**  
履歴僅少時は、ユーザープロフ（方言・一人称）で補完しつつ、相手のプロフから話題を拾う「探り」質問を生成する。fileciteturn38file0L106-L107

- `BigClient`: 安っぽい褒め、軽薄なノリ、他客（他者）との比較による優越感煽りの禁止。fileciteturn38file0L113-L114

# 付録AE: 設計に至った経緯（MUST / 仕様の意図）

**Added (JST):** 2026-03-01 23:35:00 +0900

本章は「夜職タイプ診断と分類提(確定).txt」に基づく仕様の **意図** をSSOTとして残す（後続の手戻り防止）。fileciteturn38file0L3-L120

## AE-1. 二重構造（Night×Real）にした理由

- 営業LINEの最適化は「営業スタイル（Night）」と「価値観/生活（Real）」が混ざって決まるため、単一軸では再現性が落ちる。fileciteturn38file0L7-L15
- 返信生成では Night が「狙い（来店/太客/駆け引き等）」を、Real が「言い回しの価値観（安定/自立/承認等）」を決める。

## AE-2. 返信生成で “履歴ミラーリング” を最優先にした理由

- 夜職のLINEは「相手のテンション・癖・距離感」を合わせないとAI感が出て、離脱や信用毀損に直結する。fileciteturn38file0L102-L107

## AE-3. 呼称推定を入れた理由（ただし誤認は致命）

- 名前/呼称は距離感を一気に変える強い要素。正しく推定できれば強いが、第三者名の誤認は関係破壊に直結するため、判定不能時は入れない。fileciteturn38file0L11-L15

## AE-4. 時間整合性を考慮する理由

- 夜の時間帯は「挨拶の自然さ」が体験品質に強く影響するため、端末時刻（client_now）と履歴から自然な導入を作る。fileciteturn38file0L105-L107

## AE-5. A/B/Cの役割を固定する理由

- 3案提示は「断定しない」ための中核UX。役割を固定すると、ユーザーが状況に応じて選びやすく、毎回の生成ブレも抑えられる。fileciteturn38file0L116-L120

## AE-6. 返信長さ制約を撤回した理由

- 140文字制約は短文化を誘発し、夜職の営業LINEとしては情報量/空気感が不足しやすい。
- よって「短文縛り」を外し、履歴の熱量に合わせた **長文寄り** を正とする（付録AD参照）。
