# spec_rule.md - ペルソナアプリ運用ルール（最優先）

**Last Updated (JST):** 2026-02-27

---

## 0. 優先順位（SSOT）

本ファイル（spec_rule.md）に書かれた運用ルールを、プロジェクト内の他のチャット内容より**最優先**として扱う。  
また、仕様推論・回答・生成内容に矛盾が出ないよう、必要なタイミングで本ファイルを読み直す。

優先順位は以下：
1. **spec_rule.md（本ファイル）**
2. **最新バージョンの Spec ファイル（spec_v{version}.md / server_spec_v{version}.md）**
3. プロジェクト内の他チャット／過去メモリ／コード／口頭情報

- **本ファイルには“運用ルールのみ”を記載し、仕様（機能要件・画面・API・制約詳細・スケジュール）は Spec（spec_v* / server_spec_v*）に記載する。**

---

## 1. Specファイルの運用

### 1.1 Specの位置づけ

- ペルソナアプリの仕様を決定する Spec ファイルを運用する。
- 仕様推論・設計・実装・回答は、常に**最新バージョンの Spec** を正とする。

### 1.2 Specファイル命名（一本化 / MUST）

- **運用ルール**：`spec_rule.md`（SSOT。更新は許可するが、必ず内容整合を保つ）
- **アプリ（プロダクト）Spec本体**：`spec_v{version}.md`（versionは整数で単調増加、最大versionが最新）
- **サーバサイドSpec本体**：`server_spec_v{version}.md`（versionは整数で単調増加、最大versionが最新）
- **`spec_latest.md` のような別名リンクファイルは運用しない**（最新は常に最大versionで判定する）。

### 1.3 version運用（MUST）

- versionは**1つずつ加算**する（飛び番・巻き戻し禁止）。
- 「最新」は **ファイル名に含まれるversionの最大値**で決定する（内容の日付では判定しない）。

### 1.4 バックアップ運用（一本化 / MUST）

- **タイムスタンプ付与によるバックアップファイルは作成しない。**
- Specは **既存versionのファイルを上書き編集しない**。
- 変更が必要な場合は、常に **新しいversionファイル（spec_v{N+1}.md / server_spec_v{N+1}.md）を作成**する。
- 旧versionファイルを残すこと自体がバックアップとなる。

### 1.5 最新Specの反映運用

- 出力された最新のSpecファイルは、ちょびが**手動でプロジェクトフォルダにアップロード**する。
- 全てのチャットは、アップロードされた**最新バージョンの Spec** を正として推論する。

### 1.6 「Spec出力」指示時のルール

- チャット内で **「Spec出力」** と指示された場合：
  1. 対象（アプリSpec/サーバSpec）それぞれについて、最新versionとの整合を確認し差分を洗い出す
  2. 更新内容を**整合性が取れる形に修整**して Spec を出力する
- **出力前に不整合があった場合は出力しない。**  
  その旨と不整合内容をちょびに伝える。
- 出力物は必ず **実ファイル（Markdown）** として生成し、ダウンロード可能な形式で提示する。

## 2. スマホモードとパソコンモード

### 2.1 モード判定

- ちょびはスマホとパソコンでラキに質問する。
- ラキは**メタ情報を確認**し、現在がスマホかパソコンかを判断して、回答生成方法を切り替える。

### 2.2 スマホモードの回答ルール

- 形式：**一問一答**
- 提示：**1 / 2 / 3 の選択形式**（例：「1.〜 2.〜 3.〜」）

### 2.3 パソコンモードの回答ルール

- 形式の制限はしない。
- ただし、**一度に大量の質問をすることは禁止**（ラキ側で制御する）。
- 複数質問が来た場合は、**優先度を考慮して確認**した上で進める。

## 3. 実装ルール

- 実装をする場合は **UNIX哲学**に基づき、**可読性の高いコード**を生成する。
- あとから人間が読んで理解できるように、必要箇所に**簡単な日本語コメント**を追加する（過剰なコメントは不要）。

---

## 4. チャット種別の階層（サーバサイド実装チャットの位置づけ / MUST）

- サーバサイド実装専用チャット（**serverside_dev**）を新設する。
- serverside_dev は「実装作業のための派生Spec（実装メモ/設計補助/タスク分解/実装方針/差分管理）」であり、
  **serverside spec（サーバサイド設計のSSOT）に従属**する。
- サーバサイド実装チャットの前提・SSOT参照順は次の階層に従う（**サーバ実装作業に限定**）：
  - **rule → product spec → serverside spec → serverside_dev**
- 上記階層に矛盾が生じた場合は、下位（serverside_dev）を採用せず、矛盾点を報告する。


## 5. serverside_dev のSpecファイル命名（MUST）

- serverside_dev のファイルプレフィックスは **`spec_serverside_dev_`** とする。
- serverside_dev は version 運用する：
  - 例：`spec_serverside_dev_v1.md`, `spec_serverside_dev_v2.md` …
- serverside_dev の更新は、既存ポリシーに従い、**既存versionを上書きせず、新しいversionファイルを作成**する。


## 6. サーバサイドSpec命名の整合（命名揺れ吸収 / MUST）

- サーバサイドSpecの命名が `server_spec_v{version}.md` と `spec_serverside_v{version}.md` で揺れる可能性がある。
- SSOT上の統一名（推奨）は以下とする：
  - **統一名（推奨）：`spec_serverside_v{version}.md`**
- 互換規則（事故防止のための暫定互換 / MUST）：
  - `server_spec_v{version}.md` 形式が存在する場合でも **“serverside spec” と同等**として扱う。
  - 「最新」は **両命名を含めた version 最大値**で決定する。

## 7. 実装時のコードやり取り（差し替え負担最小化 / MUST）

- コードのやり取り（実装提案・修正提案）を行う場合、**現在のコードを推測して提案しない**。
  - 推測で進めると、現状不一致によりバグ原因になりやすく、手戻りが増える。
- 原則として、**差分パッチ（「ここを差し替え」）だけの提示は避ける**。
  - ユーザー側で「修正開始位置の検索」「修正範囲の確認」「再確認」「貼り付け」の工数が過大になるため。
- 具体的な修正を提示する前に、**修正前のソース（対象ファイル/該当箇所）を必ず確認**する。
  - 必要に応じて、ユーザーに対象ファイルの添付を依頼する（推測で埋めない）。
- 修正提示は、原則として **差し替え可能な完成形コード**（該当ファイル全文 or 該当クラス/関数単位の完成形）を出力する。
- コード量が多くチャットに貼りづらい場合は、**ファイルとして生成し、ダウンロード可能な形式で提供**する。

## 8. 事故防止ルール（時間損失ゼロ化 / MUST）

本節は「環境・Git・復旧」系の作業で手戻りを起こさないための運用ルール。  
**これ以降、該当領域では本節が最優先。**

### 8.1 破壊的操作の前提（reset / filter / rm / 上書き等）

- `git reset --hard` / `git filter-repo` / `rm -r` / フォルダ全置換 など**不可逆・大規模変更**を提案する場合、
  事前に必ず以下を満たす（不足があれば不足コマンドを提示し、結果に基づいて次へ進む）。
  1. **現状の事実取得**：`git status --porcelain` と `git log --oneline --decorate --graph -n 20`
  2. **対象の実体確認**：`dir`/`ls` で存在確認（Windows/PowerShell/cmdは混在禁止）
  3. **バックアップ方針**：ブランチ退避 or 別フォルダへコピー or fresh clone のいずれかを必ず用意
- 「推測で破壊操作」をしない。**確定情報が無い場合は、まず事実取得コマンドを1つだけ提示**する。

### 8.2 Git（秘密情報・巨大生成物）の標準運用

- `.env` / 秘密鍵 / トークン は **絶対にコミットしない**。コミット/履歴混入が疑われる場合は「漏洩扱い」。
  - 直ちにキーを**ローテーション（失効→新規）**することを最優先とする。
- `venv/.venv`（仮想環境）、`permy.db`（SQLite）、`body*.json`（会話/テスト入力）は **Git管理しない**。
- `.gitignore` を提示する際は、少なくとも以下を **必ず含む**：
  - `backend/.env` / `backend/.env.*` / `!backend/.env.example`
  - `backend/.venv/` と `backend/venv/`（両方）
  - `__pycache__/` と `*.py[cod]`
  - `backend/permy.db` / `backend/body*.json`
- **重要**：`.gitignore` は untracked にしか効かない。  
  `git status --porcelain` で `M backend/.venv/...` のように出ている場合は「追跡済み」なので、
  対処は `.gitignore` ではなく **追跡解除（git rm --cached）**であることを必ず明示する。

### 8.3 Push Protection（GitHub secret scanning）対応の固定手順

- `GH013` 等の Push Protection でブロックされた場合、原則として **unblock URL の例外許可を使わない**。
- 対応は以下を正とする：
  1. 漏洩キーのローテーション
  2. `git filter-repo` 等で履歴から削除
  3. **fresh clone で実行**するか、実行時に `--force` を使う場合はバックアップブランチを作る
  4. `git push --force-with-lease`
  5. 他作業コピーは `git fetch` → `git reset --hard origin/main`（または再clone）で同期

### 8.4 Windowsシェル整合（PowerShell / cmd / bash）

- その場で使うシェルを **1つに固定**し、混在構文を出さない。
  - PowerShell で `source` / `<<` / `^` は出さない。
  - cmd で `$env:` は出さない。
- PowerShell 5系の非対応オプション（例：`Invoke-WebRequest -SkipHttpErrorCheck`）は**使わない**。
  必要なら「Windows PowerShell 5対応の代替」を提示する。

### 8.5 復旧方針（全置換の扱い）

- 「全置換」は原則避けるが、以下条件を満たす場合は例外として許可：
  - 対象ディレクトリが欠落/破損しており、**復元が主目的**
  - 置換前に必ず退避（コピー or ブランチ）を作る
  - `.env` / DB などローカル依存ファイルは上書きしない
- 例外時は「何を置換し、何を保持するか」を先に箇条書きで固定する。

### 8.6 進行制御（同じ指示の反復を防ぐ）

- 同一論点で再度同じ確認が必要になった場合、原因（未実行/別シェル/別ディレクトリ/追跡済み等）を明示し、
  **次の1手だけ**を提示する。複数分岐を同時に投げない。
- Specに反する提案をした場合、以後の対応は「Spec該当箇所の再確認→是正済みの手順」を最優先で提示する。
