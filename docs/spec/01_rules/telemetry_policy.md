# Telemetry Policy（本文ゼロ・時間帯分析対応 / MUST）

**Last Updated (JST):** 2026-03-02 04:35:00 +0900

## 1. 目的（MUST）

- **使用時間帯（ピーク/谷）**を把握し、将来のサーバ移行・リプレース計画に活かす
- 品質改善（失敗率/レイテンシ/NGゲート発火率/followup発生率）
- 収益/プラン分析（free/pro/premium比率、上限到達率）

## 2. 最上位制約（絶対 / MUST）

- **会話本文（トーク履歴txtの中身）を送らない・保存しない**
- **生成本文（返信案本文）を送らない・保存しない**
- 個人情報（氏名/連絡先/住所等）をログに含めない
- デバッグ目的で本文をログに出す行為を禁止（例外なし）
- OpenAI API の **リクエスト/レスポンス生ペイロードをログに出さない**

## 3. 時刻の扱い（現時点確定 / MUST）

- 分析の正は **サーバ時刻（UTC）** とする
  - `server_time_utc` を必ず付与
  - `hour_bucket_utc`（0..23）と `dow_utc`（0..6）をサーバで算出し保存
- クライアント時刻 `client_now` は、UIの自然さ用途（挨拶等）のみで利用可
  - **分析基準にしない**

## 4. 収集粒度（現時点スコープ / MUST）

- 時間帯は **UTCの時間バケットのみ**
  - `hour_bucket_utc`（0..23）
  - `dow_utc`（0..6）
- 将来の拡張（15分バケット等）は、ユーザー明示指示があるまで実施しない

## 5. 保存方式（MUST）

- **生ログ**（イベント）は短期保持（推奨 7〜30日、運用で決める）
- **長期保持**は集計（hourly rollup）のみ
  - 生ログを長期保持しない（個人特定リスクと運用負荷を下げる）

## 6. 識別子（MUST）

- アプリ内部の匿名ユーザーIDは分析基盤に直接流さず、サーバ側で **HMACハッシュ化**した `user_id_hash` を使用する
- `user_id_hash` は復号不能（ソルト/鍵管理はサーバ側）

## 7. 監査（MUST）

- 収集フィールド追加は必ずPRで提案し、ちょび承認後に実施する
- 本文ゼロ制約を破る変更は一切禁止
