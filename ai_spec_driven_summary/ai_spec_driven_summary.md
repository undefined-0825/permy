# AI開発の現状

現在のAI開発は多くの場合**AIを補助ツールとして使用**という形です。

| 項目 | 内容 |
|------|------|
| 仕様管理 | 人間 |
| 実装 | 人間 + AI |
| 成功要因 | 個人スキル |
| 再現性 | 低い |

この方法でも成果は出ますが**属人性が高い**という問題があります。

------------------------------------------------------------------------

# 今回のアプローチ

今回考案したのは

**Spec駆動型AI開発**

です。

| 項目 | 内容 |
|------|------|
| 仕様管理 | AI |
| 実装 | 100% AI |
| 人間の役割 | 設計のみ |
| 再現性 | 高い |

つまり

**人間は設計に集中し、ドキュメント管理や実装はAIに任せる**

という開発手法です。

これは私の長年の経験で培われた経験則を前提にしていますが、
- 設計の甘いプロジェクトは破綻する
- ドキュメントは次第に陳腐化していく(もしくは行方不明になる)
- 実装者のスキルが低いプロジェクトは品質が下がる
- 人間には体力・精神的に限界がある
- 経験不足のPM/SEだと考慮漏れが多発する

という点です。

私も50歳になったので、体力面は落ちています。

ただ、この開発は帰宅してYoutubeを見ながら
- 「次のtodo」
- 「ok」
- 「commit」
- 「specに反映」
- 「実装とspecの差異をチェック」

を繰り返し打ってしていっただけなので、体力なくても十分できるかなと思っています。

【Spec駆動】

# 長期メモリ

AIは膨大な知識を持っていますが

-   高品質な知識
-   一般レベル情報
-   不正確な情報

が混在しています。

この中で本当に欲しい情報は「高品質な知識」のみです。

設計はChatGPTの長期メモリを使い

-   私の思考レベル
-   私の期待する出力形式
-   私の設計思想や何を重要としているか
-   私がどのレベルの議論を期待しているか

を定義することで「**AIの知識をフィルタリング**」しました。

つまり、一般的な知識とか、Webの検索結果をつなぎ合わせて成形しただけの当たり障りのない回答は不要。

これをするだけで、AIはかなり変わります。
これにより、私とAIとの壁が少なくなり、設計のスピードが大幅に向上しました。

さらにChatGPTの「プロジェクト機能」を活用し、spec設計やルール決定を厳密化してみました。

一般的にはChatGPTよりGemini等のAIがもてはやされていますが、私にとってはフィルタリング可能なChatGPTが、今のところ推論ベースでは一番使えるAIと感じています（最新の情報には疎いけど）

AIはただの推論ツールなので、カスタマイズしてこそ本当の力が発揮できると思います。

# スマホモード

今回の2カ月は稼働が高く平日はほぼ開発できませんでした。

そこで「**スマホモード**」という概念ををAIに覚えさせてみました。

**メタ情報からAIが自動判定**し、どのデバイスか認識してモードを自動で切り替えます。

スマホモードの場合は片手＋狭いディスプレイでも即座に反応できるように、

| 項目 | 内容 |
|------|------|
| 回答形式 | 一問一答 |
| 意思決定 | 1, 2, 3（選択肢にない場合は入力） |
| 文章量 | 最小 |

設定したことで、2時間の通勤時間を設計に充てることができました。

音声入力とかも活用しています。

考えがまとまらない場合はスマホに思いつくままに喋ってまとめさせる。
ただ、気を付けないと変な50歳がブツブツとスマホに語りかけるというホラーになるので、家とか人のいない時しか使えない必殺技です。

# Copilot暴走事件

ある日、いつものように「実装！」と指示を出してYoutubeを見ていたら、AIが何か延々実装をしている。
ログを見てみると

①依存関係エラー\
↓\
②AI自動修正\
↓\
③OSSバージョン変更\
↓\
④環境破壊\
↓\
⑤commit連打（①に戻る）

気が付いた時には、リポジトリ汚染による破壊（短期間の修復は不可能な状態）

になってしまい、**2回目のプロジェクトも破棄**になりました。

# AIに何をさせないか

一般的なAI活用は「**AIで何ができるか**」を考えます。

しかし今回の開発では「**AIに何をさせないか**」が重要ということがわかりました。

今までの経験でも「**プログラマに自由を与えると、いい結果にならない**」ということは自分の中では一貫していた経験則なので、AIに制約をかける方向へ修正。


------------------------------------------------------------------------

# AI制御

AI暴走防止のため

  制約           内容
  -------------- ----------------
  破壊操作禁止   reset / filter
  品質ゲート     テスト成功
  commit制御     自動commit制限(github側にも制限)

を導入しました。
テスト駆動は初期の段階から導入していたが品質ゲートを設けることでAIの手抜き(テスト失敗しているのに完成したと報告する)を防ぎました。
長期的な改修になりそうな場合は事前に報告が必要とか、修正で困ったら中断するとかさせています。

# AI開発の4レイヤーモデル

今回の方法論を整理すると

AI開発は以下の4レイヤーで構成されます。

  レイヤー           内容
  ------------------ ------------------------------------
  Layer1 Knowledge   LLMの知識
  Layer2 Filter      長期メモリによる知識フィルタリング
  Layer3 Spec        Spec駆動開発
  Layer4 Control     AI制御ルール

# 組織へのインパクト

この方法が確立すると

  領域       影響
  ---------- --------------
  PoC        開発速度向上
  MVP        コスト削減
  新規事業   試行回数増加

つまり

**アイデアの実験速度が上がる**

という効果があります。

さらに開発時点で技術者側のスキルは0でもできるという点です。

- 「このアイデアを形にする場合に最適なフレームワークは？」
- 「国内での開発事例は？資料は豊富にある？」
- 「次点のフレームワークと比較してメリデメは？」
- 「俺一人で開発するとどれくらいの工数？」
- 「このシステムを作った場合、今のspecで足りない要素は？」
- 「個人情報保護の観点からみて考慮漏れのポイントがある？」
- 「●●という機能を実装しようと思うが戦略的にあり？」
- 「このコンセプトを変えようと思うけどどうかな？開発工数のびる？」
- 「将来的な戦略を考えるためにユーザ分析をしたい。●●の情報を統計的に収集するように変更」
- 「●●への導線は一般的なユーザが自然に理解できる？」
- 「ここまでの議論を最新specと比較して差分出力」
- 「そこじゃねーだろ！何度言えばわかるんだ！spec読み直せ！で、今指摘されたことを自己分析して、二度と繰り返さないようにspecに追加しろ！！！」
　　⇒AIには感情なんてものはこれぽっちもないので何を言っても無駄です。
　　　ただ、口調とかは理解できるので厳しい口調で指摘するとヤバイと思ってかなりいい感じにspecを修正します（笑）

みたいな感じで進めました。

- AIは与えられたことを整理してまとめていく力が優れている
- 逆にアイデアを0から創造することや、選択肢が多い場合は苦手分

指示(spec)がしっかりしていれば確実に人間より高速な実装ができるので、これを最大限に引き出すためにspecの厳格化とAIが迷わないための指針(目的・ゴール、優先度の指標)をしっかりとspecに落とし込んでおくことが重要になります。



- 設計者は頭の中にあることをどんどん出していく。
- AIが案を出してくるので決定して形にする。
- 最終的なspecの更新はAIにさせる。

これで時系列がバラバラでも整合性の取れるspecを作成することができます。


今回の実験は、私が未経験のスマホアプリ開発を題材として進めてみましたが、実績のある実務アプリやWeb開発等ではさらに効率的に設計ができると思います。

- product, frontend, backendのspecをそれぞれのプロジェクトに合ったspecに差し替えていけば様々な開発に応用できると思います。

今後の展開案としては

- 顧客とのミーティングをTeams録画して文字起こしした要件定義からspecを展開していけばヌケモレもなくPoC作成
- アジャイルの弱点である頻繁な仕様変更によるメンバーの疲弊にも対応可能
- 現行の開発段階のプロジェクトもまずAIにコードからspecを整理させることで改善できるかもしれません

# なぜ社内公開なのか

このspec駆動型開発手法は個人の技術ブログで公開すれば広く拡散する可能性があります。

しかし今回は**社内限定で共有**します。

理由は

この手法が確立すると

NJK全体として

-   開発工数削減
-   新規事業開発の高速化

など大きなインパクトがある可能性があるためです。

- 新人でもアイデアさえあればプロジェクトが進めれる（未経験でもやり方さえ理解できればできる）
- 短期間で目に見える実績が体感できる（AIの生成待ちは他の作業ができる）
- コーディングはほぼ不要（ただし、AIが生成するコードを判断するスキルは必要）
- 他社との明確な差別化ができる(NJKの品質は細部まで設計されていて良い等)

各プロジェクトのプロンプトやspecを見てみましたがまだまだ改善する余地はあるので、一つの検証結果として参考にしてもらえればと思います。
